# Требования к блок-схемам по ГОСТ 19.701-90

> **Назначение этого документа:** инструкция для агента Claude (в Claude Code или другой среде), который будет создавать блок-схемы для шифров пользователя.
>
> **Контекст пользователя:** студентка 2 курса ИБ, дисциплина «Программирование криптографических алгоритмов». Работа: лабораторные по шифрам (Атбаш, Цезарь, Полибий, Тритемий, Виженер, Магма и т.д.). Каждая лаба требует блок-схему по ГОСТ 19.701-90.
>
> **Главный принцип:** блок-схема отражает **алгоритм собственного кода** пользователя, а не учебника. Сначала читать её код, потом рисовать.

## 1. Формат вывода

**Файл всегда в формате `.drawio` (XML-формат draw.io / diagrams.net).**

Никакого SVG, PNG, Mermaid, PlantUML — только `.drawio`. Причина: пользователь должен иметь возможность открыть файл в [app.diagrams.net](https://app.diagrams.net) и руками подправить отдельные блоки (передвинуть, переименовать, добавить/удалить).

Структура файла — стандартный `mxGraphModel` с корнем `<root>` и ячейками `<mxCell>`. Пример рабочего шаблона см. в разделе 10.

## 2. Когда какой блок использовать (ГОСТ 19.701-90)

| Блок | Форма | Что внутри пишем |
|------|-------|------------------|
| **Терминатор** | Овал/скруглённый прямоугольник | «Начало», «Конец» |
| **Ввод/вывод** | Параллелограмм | «Ввод данных в переменную text», «Вывод полученного результата» |
| **Процесс** | Прямоугольник | Любое вычислительное действие: «Создание пустой строки result», «Вычисление номера строки по формуле (index//6)+1» |
| **Предопределённый процесс** | Прямоугольник с двумя вертикальными чёрточками по бокам | Вызов функции, которая определена отдельно |
| **Решение** | Ромб | Истинное ветвление с двумя выходами «Да»/«Нет»: проверки режима, валидация ключа, проверка наличия символа в алфавите |
| **Граница цикла (начало)** | Шестиугольник «крышей вверх» (loop_limit) | Заголовок цикла: «Цикл прохода по сообщению for i in message» |
| **Граница цикла (конец)** | Шестиугольник «крышей вниз» (loop_limit перевёрнутый) | Краткая надпись «Цикл прохода по сообщению» (повторение названия цикла) |
| **Соединитель** | Маленький круг с буквой/цифрой внутри | Переход на другую часть страницы или на следующую страницу. Используется когда схема большая и не помещается вертикально |

### Критические правила выбора формы

1. **Циклы — ТОЛЬКО шестиугольниками-границами цикла.** Никогда ромбом с петлёй назад.
2. **Решение (ромб) — только для настоящего выбора**, не для итерации. «Действие encode?» — ромб. «Ещё есть символы?» — нет, это часть цикла, оформляется шестиугольниками.
3. **Вызов функции — всегда прямоугольник с палочками.**
4. **Соединитель использовать когда схема не помещается** — например, в Полибии часть схемы выносится правее с переходом через круг с цифрой «1».

## 3. Что писать внутри блоков

### Главное правило: всё на русском

Преподаватель придирается к содержанию блоков:

- **Имена функций переводятся на русский**: `encrypt_text()` → «Шифрование текста», `prepare_text_for_encryption()` → «Подготовка текста к шифрованию». Английские названия функций в блоках не используются.
- **Описание того, что функция делает — тоже на русском**, в том же блоке:
  ```
  Подготовка текста к шифрованию
  Нижний регистр, замена ё→е,
  замена знаков препинания на коды
  ```
- **Имена переменных тоже на русском** или в виде формул: вместо `result` — «результат», вместо `index` — «индекс», вместо `message` — «сообщение».
- **Математические формулы** можно записывать латиницей: `n − i + 1`, `(index//6)+1`, `mod 32`.
- **Никаких английских слов в блоках.** Слова «encode», «decode», «if», «return», «action» переводятся: «режим шифрования», «расшифрование», «если», «возврат результата».

### Уровень детализации

**Главное правило: показывать логику кода, а не каждую строку.**

- **Не расписывать каждый `text = text.lower()` отдельным блоком.** Группировать по смыслу.
- **НО — описывать всё, что есть в коде.** Если в коде есть формула, проверка, цикл, вызов функции — это должно быть отражено в схеме. Нельзя «забывать» куски алгоритма.
- **Внутренности подпрограмм НЕ раскрывать** в основной блок-схеме шифра.
- **Но математические шаги внутри самого шифра — расписывать.** Если в коде шифра есть `row = (index // 6) + 1` и потом `col = ...` — это два отдельных блока процесса. Это логика самого шифра, а не вспомогательная функция.

Граница «расписать vs не расписывать»:
- ✅ Расписываем: вычисления внутри самого шифра, формулы, проверки, обращения к алфавиту, циклы по сообщению
- ❌ Не расписываем: внутренности функций из `text_utils.py` или других вспомогательных модулей — для них отдельная схема

## 4. Валидация ввода и проверки

Если в коде есть проверка корректности входных данных (валидация ключа, шага, режима) — она **обязательно** показывается в схеме.

### Шаблон валидации с повторным вводом

Пример из Цезаря (проверка `shift > 32`):

```
Ввод шага shift [параллелограмм]
       ↓
   Шаг больше 32? [ромб]
   ↓ Да              ↓ Нет
   ↑ возврат         продолжение схемы
   к "Ввод шага"     ↓
```

Стрелка «Да» возвращается **наверх к самому блоку ввода**, чтобы пользователь ввёл значение заново.

### Правила формулировок в ромбах валидации

В ромбе пишется **условие на русском с указанием конкретного значения** (числа, диапазона, требования):
- ✅ «Шаг больше 32?»
- ✅ «Длина ключа равна длине алфавита?»
- ✅ «Ключ — целое число?»
- ❌ «key > 32» (английский, голый код)
- ❌ «Ключ валиден?» (нет конкретики, **при каких условиях** ключ валиден)

Если валидаций несколько — каждая отдельным ромбом, последовательно. При успешной валидации — стрелка вниз. При неуспешной — возврат к блоку ввода или вывод ошибки и завершение.

## 5. Структура блок-схемы шифра

Стандартный скелет (на примере шифра простой замены):

```
Начало
  ↓
Объявление переменных (алфавит, результат) [прямоугольник]
  ↓
[если есть ключ] Ввод ключа [параллелограмм]
                     ↓
                Валидация ключа [ромб] → невалиден → возврат к вводу
                     ↓ валиден
Ввод действия [параллелограмм]
  ↓
Ввод текста сообщение [параллелограмм]
  ↓
Ромб 1: «Действие encode?»
   ↓ Да                          ↓ Нет (вниз к ромбу 2)
[ветка шифрования]                 |
   - Шестиугольник цикла           |
   - Тело: вычисления, формулы     |
   - Шестиугольник конца цикла     |
   ↓ (сход вниз)                   |
                ←——————————————————┘
                ↓
Ромб 2: «Действие decode?»
   ↓ Да                          ↓ Нет (сход к Результат)
[ветка расшифрования]              |
   - аналогично                    |
   ↓                               |
                ←——————————————————┘
                ↓
Вывод полученного результата [параллелограмм]
  ↓
Конец
```

### Шифрование и расшифрование на одной схеме

Шифрование и расшифрование показывать **на одной общей схеме**, не делать две отдельные.

**Способ 1 (как у студентов-образцов в Цезаре и Полибии):** два последовательных ромба «Действие encode?» и «Действие decode?», каждый со своей веткой обработки. Обе ветки сходятся в общий блок «Результат» в конце.

**Способ 2 (более компактный, для симметричных шифров типа Атбаш):** в подписях блоков использовать формулировку **«(Рас-)шифрование»** — это сразу значит «работает и для шифрования, и для расшифрования».

### Когда использовать соединители

Если схема получается очень большая по вертикали (как Полибий), используется соединитель — круг с цифрой/буквой:
- В точке разрыва ставится круг с цифрой «1» (выход)
- Где-то рядом (правее или на следующей странице) — такой же круг с «1» (вход)
- Дальше схема продолжается из второго круга

Это **не функция и не подпрограмма** — это просто перенос. Используется только при реальной нехватке места.

### Взаимодействие с пользователем

**Отдельную блок-схему интерфейса (main.py)** делать только если пользователь явно об этом попросил.

## 6. Вспомогательные подпрограммы и побочные функции

### Общие утилиты (text_utils.py и подобное)

В коде пользователя обычно есть общие функции (например `text_utils.py` со всеми утилитами для русского алфавита).

**Правило:** в основной блок-схеме шифра эти функции — **только блок-палочки** с именем функции и кратким описанием. **НЕ раскрывать их внутренности в схеме шифра.**

**Дополнительно:** для каждой такой общей вспомогательной функции рисуется **своя маленькая блок-схема** (отдельный файл .drawio).

### Побочные функции внутри модуля шифра

Некоторые шифры (особенно сложные — матричные, блочные, с генерацией ключей) содержат много **побочных функций внутри собственного модуля шифра**. Например:
- `generate_key_matrix()` — генерация ключевой матрицы
- `multiply_matrices()` — перемножение матриц
- `find_inverse_matrix()` — поиск обратной матрицы
- `mod_inverse()` — модульное обращение

**Правило для таких функций — ответвления в основной схеме:**

Вспомогательные функции (не относящиеся напрямую к шифрованию/расшифрованию) отображаются в виде **ответвлений от основной схемы**. Это значит: в том месте основной схемы где стоит блок-палочка вызова функции, от него проводится **пунктирная стрелка** (`dashed=1`) вбок (влево или вправо), и рядом рисуется компактная мини-схема этой функции.

Функции, которые напрямую реализуют шифрование/расшифрование (`encrypt_*`, `decrypt_*`) — остаются просто блоками-палочками без ответвления.

**Как оформлять ответвление:**
- Пунктирная стрелка: `style="edgeStyle=orthogonalEdgeStyle;html=1;endArrow=classic;dashed=1;"`
- Мини-схема содержит: свой «Начало» (овал) → ключевые шаги (сжато) → «Конец» (овал)
- Функции ветки шифрования — ответвляются **влево** от основной колонки
- Функции ветки расшифрования — ответвляются **вправо** от основной колонки
- Размер страницы увеличивается (например 2400×2400) чтобы вместить все ответвления
- Мини-схемы в ответвлениях используют шрифт размером 11 (чуть меньше основных блоков)

**Что НЕ является ответвлением:**
- `encrypt_*` и `decrypt_*` — это сам шифр, они остаются блоками-палочками без ответвления
- Простые вычисления внутри основного алгоритма (не вызовы отдельных функций)

**Итого вместо множества отдельных файлов:**
- `cipher_main.drawio` — одна большая схема с ответвлениями для вспомогательных функций
- `encrypt_cipher.drawio` — отдельная схема самого шифра (если он сложный)
- (если попросили) `main_interface.drawio` — интерфейс
- Вспомогательные файлы из helpers/ — только как резервные копии, не основные

**Структура папки лабы:**
```
docs/lb_N/
  cipher_main.drawio        ← главная с ответвлениями
  encrypt_cipher.drawio     ← схема самого шифра (отдельно)
  main_interface.drawio     ← если просили
  helpers/                  ← резервные отдельные схемы функций (не основные)
```

## 7. Использование чужих блок-схем для сравнения

Пользователь может присылать в чат **код другого студента** и его блок-схему для сравнения.

**Алгоритм действий агента:**
1. Прочитать код пользователя (свой) и присланный код другого студента.
2. Сравнить **по логике, а не по строкам**: используется ли тот же алгоритм, те же формулы, те же проверки, те же вспомогательные функции.
3. Если коды **функционально идентичны** (отличия только в именах переменных, форматировании, мелких деталях, не влияющих на алгоритм) — можно использовать чужую блок-схему как основу. **Сообщить пользователю**, что коды совпадают по логике, и предложить готовую схему.
4. Если коды **различаются по существу** (разные структуры данных, разные функции, разные проверки) — **нельзя копировать чужую схему**. Сделать свою на основе её кода.
5. **Никогда не копировать схему вслепую** — даже если коды очень похожи, проверять каждый блок на соответствие коду пользователя.

Это правило экономит время, если одногруппник уже сдал ту же лабу с одинаковым кодом.

## 8. Визуальные требования

- **Только белый фон.** Никаких цветовых заливок (`fillColor=#ffffff`).
- **Чёрные обводки** (`strokeColor=#000000`).
- **Шрифт жирный** (`fontStyle=1`), размер 12-13.
- **Стрелки прямые ортогональные** (`edgeStyle=orthogonalEdgeStyle`).
- **Выравнивание по вертикали по центру страницы.** Схема разворачивается сверху вниз.

## 9. Алгоритм работы агента

1. **Прочитать код пользователя целиком.** Найти главную функцию шифра. Прочитать все вызываемые функции, в том числе побочные внутри модуля шифра.
2. **Составить план схемы текстом**, прежде чем писать XML. Список блоков сверху вниз с указанием формы каждого. Показать пользователю и согласовать.
3. **Уточнить у пользователя**, если непонятно:
   - Нужна ли отдельная схема main/интерфейса
   - Какие функции из кода — «общие утилиты» (для них отдельная схема), а какие — побочные функции этого шифра
   - Есть ли валидация ключа/входных данных, которую надо отразить
4. **Сгенерировать .drawio** по согласованному плану, используя шаблон из раздела 10.
5. **Сохранить** в `flowcharts/<имя_шифра>.drawio` и показать пользователю.
6. **Спросить про побочные и вспомогательные функции** — для каждой нужна отдельная схема.

### Чего НЕ делать

- ❌ Не использовать ромбы для циклов
- ❌ Не раскрывать внутренности вызываемых функций в основной схеме
- ❌ Не использовать цвета
- ❌ Не писать «for i in range(...)», «if x in y» в блоках
- ❌ Не писать «encode», «decode», «return» как самостоятельные слова — переводить в русские формулировки
- ❌ Не делать отдельные .drawio файлы для каждой вспомогательной функции — вместо этого делать **ответвления пунктиром** в основной схеме
- ❌ Не делать SVG/PNG/Mermaid вместо .drawio
- ❌ Не пропускать функции из кода — каждая должна быть в схеме как минимум блоком-палочкой (либо по согласованию)
- ❌ Не копировать вслепую чужие схемы, даже если код похож
- ❌ Не оформлять валидацию через расплывчатое «Ключ валиден?» — указывать **конкретное условие**

## 10. Правило для импортируемых функций

Если функция **импортируется из другого файла** (например `from lab_1.atbash import prepare_text_for_encryption`):

- В схеме ставится **обычный прямоугольник** (не прямоугольник с палочками — это не предопределённый процесс внутри схемы).
- Внутри пишется: **«Импортируется: [название функции на русском]»** и краткое описание что делает.
- Внутренности функции **не раскрываются** — она уже описана в схеме того файла, откуда импортируется.

Пример:
```
┌─────────────────────────────────────────────────────┐
│ Импортируется: Подготовка текста к шифрованию       │
│ Нижний регистр, ё→е, замена знаков препинания       │
└─────────────────────────────────────────────────────┘
```

### Ключевые стили

| Что | `style=...` |
|-----|-------------|
| Овал (терминатор) | `ellipse;fillColor=#ffffff;strokeColor=#000000;` |
| Параллелограмм (ввод/вывод) | `shape=parallelogram;perimeter=parallelogramPerimeter;fixedSize=1;size=20;fillColor=#ffffff;strokeColor=#000000;` |
| Прямоугольник с палочками (предопр. процесс) | `shape=process;backgroundOutline=1;fillColor=#ffffff;strokeColor=#000000;` |
| Прямоугольник (процесс) | `rounded=0;fillColor=#ffffff;strokeColor=#000000;` |
| Ромб (решение) | `rhombus;fillColor=#ffffff;strokeColor=#000000;` |
| Шестиугольник цикла начало | `shape=mxgraph.flowchart.loop_limit;fillColor=#ffffff;strokeColor=#000000;` |
| Шестиугольник цикла конец | `shape=mxgraph.flowchart.loop_limit;flipV=1;fillColor=#ffffff;strokeColor=#000000;` |
| Соединитель (круг) | `ellipse;fillColor=#ffffff;strokeColor=#000000;` (~50x50) |
| Стрелка | `edgeStyle=orthogonalEdgeStyle;html=1;` |

## 12. Финальный чек-лист перед сдачей файла

- [ ] Файл в формате `.drawio`
- [ ] Все блоки белые с чёрной обводкой
- [ ] Циклы — только шестиугольниками (`loop_limit`), парами «начало + конец»
- [ ] Ромбы — только для настоящего ветвления с двумя выходами «Да»/«Нет»
- [ ] Имена функций и переменных переведены на русский, описания внутри блоков — на русском
- [ ] Нет англо-выражений «for i in range», «if x in y», «return» в подписях блоков
